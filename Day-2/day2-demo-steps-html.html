"Let me explain how this authentication works. We created a 'robot account' in GCP — called a service account. We gave it exactly 3 permissions: deploy to Cloud Run, push Docker images, and act on behalf of services. Then we downloaded its key file and stored it as a GitHub secret. Now GitHub Actions can authenticate with GCP automatically. No human password. No manual login. The robot does it."
💬
"So the flow is: I push code → GitHub runs 32 tests → if all pass → builds Docker → pushes to Artifact Registry → deploys to Cloud Run. One git push. Everything automated. Bad code never reaches production because the tests catch it first."

Let's Break the Pipeline — Drift Detection
7:10 — 7:40
💬
"The model is deployed. It's serving predictions. But what happens when new patient data looks completely different from what the model was trained on? The model will give garbage predictions and nobody will know. Unless we check for it. That's called data drift."
✅
RUN Inference pipeline with clean data — watch it PASS
python src/inference/pipeline.py --incoming data/processed/test.csv
Watch: Step 1 loads data → Step 2 drift check passes (30% ≤ 30%) → Step 3 scores 851 records → Step 4 writes Excel
📊
OPEN the Excel output
start output\scored_*.xlsx
Click Summary sheet — shows drift result, total records, stroke count. Click Scored Results — 851 rows, color-coded predictions.
💬
"Clean data passed drift check. 851 patients scored. Excel report ready. Now let me show you what happens when bad data comes in."
❌
RUN Inference pipeline with drifted data — watch it HALT
python src/inference/pipeline.py --incoming data/incoming/drifted_patients.csv
Watch: Step 2 drift check FAILS (50% > 30%) → ⛔ PIPELINE HALTED. No predictions made. No Excel generated.
📈
OPEN the Evidently drift report
start reports\drift_report_*.html
Scroll through — per-feature drift analysis. Age shifted, glucose spiked, BMI changed. The system caught it.
💬
"The system protected itself. Bad data came in. The pipeline detected it was different from training data. It refused to make predictions. No garbage output. No false alerts to doctors. The system called for help instead of guessing."
💬
"In production, this runs automatically. A file lands in the cloud storage bucket. A Cloud Function triggers. Drift is checked. Only clean data gets scored. The Excel appears in the output bucket. No human in the loop."

Let's Talk Infrastructure as Code
7:50 — 8:00
💬
"Everything I set up in GCP — Cloud Run, storage, service accounts, alerts — I did it with manual commands. That's fine for a demo. But in production, you'd never do that. Let me show you why."
📂
OPEN terraform/ folder in VS Code
7 files. Each one replaces a manual gcloud command.
📝
CLICK through key files
main.tf — "Which cloud, which project, which APIs"
cloud_run.tf — "Creates the exact same Cloud Run service. Scale to zero. 1Gi memory."
iam.tf — "Creates the robot account with 3 permissions. Exactly what we did manually."
monitoring.tf — "Creates alert policies. Error rate > 5% → email. Latency > 2s → email."
outputs.tf — "After apply, Terraform tells you the URLs, bucket names, everything."
💬
"One command: terraform apply — creates all 13 resources. Cloud Run, storage, service accounts, alerts, everything. One command: terraform destroy — removes everything. Cost drops to ₹0. That's infrastructure as code. Reproducible, auditable, version-controlled."
