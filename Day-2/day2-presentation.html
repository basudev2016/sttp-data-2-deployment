Where We Left Off

Yesterday, we built a complete ML system — from raw data to a working API. We trained 28 models, tracked every experiment, served predictions through a dashboard, and automated testing with CI. But it all ran on my laptop.

⚠️ The Laptop Problem

If I close my laptop, the API dies. If my hard drive fails, the model is gone. If someone in another city needs a prediction, they can't reach it. A model on a laptop is a demo. Not a product.

Today, we fix every one of those problems. We'll package it, ship it to the cloud, automate deployment, add drift detection, and make the system watch itself.

Containerization: Docker

The first step to leaving the laptop behind is packaging. Docker takes our code, our model, our dependencies, and our UI — and seals them into a single, portable container. This container runs identically on my laptop, on your laptop, and in the cloud.

png1: Docker Desktop showing stroke-mlops:v2 image

What's Inside the Container

📁 Application Code

FastAPI server, preprocessing pipeline, all Python modules

🧠 Trained Model

The champion model exported from MLflow — 490KB pickle file

🌐 Dashboard UI

HTML + JS + CSS — the same interface you saw yesterday

📦 Dependencies

Python 3.11, scikit-learn, pandas, FastAPI — all locked versions

The teaching point: "It works on my machine" is no longer an excuse. The container IS the machine. Ship the container, ship the entire environment.

The Robot Account

GitHub Actions needs to talk to GCP. But we can't put a human password in a CI pipeline. So we create a service account — a robot with an ID badge and exactly the permissions it needs. Nothing more.

png9: GitHub Secrets — 3 secrets stored securely

How It Works

1. Create Robot Account

A service account named "github-deploy" — exists only for CI/CD

2. Give 3 Permissions

Deploy to Cloud Run, push Docker images, act as service account. Nothing else.

3. Store Key as Secret

The key file is stored in GitHub Secrets. The pipeline reads it at runtime. Never exposed in code.

Data Drift Detection

The model is deployed. It's serving predictions. But what happens when the patients walking in today look nothing like the patients from the training data? The model will silently give garbage predictions. Unless we check.

✅ Clean Data — Pipeline Passes

851 test records checked against reference data. 3 out of 10 features drifted (30%). Within threshold. Inference proceeds. Excel generated.

png10: Clean data passes drift check → 851 records scored

❌ Drifted Data — Pipeline Halts

Deliberately modified data: ages shifted +20 years, glucose doubled, everyone smokes. 5 of 10 features drifted (50%). Exceeds 30% threshold. Pipeline HALTED. No predictions made.

png11: Drifted data blocked — pipeline halted before any bad predictions

Infrastructure as Code: Terraform

Everything we set up in GCP — Cloud Run, storage buckets, service accounts, alert policies — we did manually. That works for a demo. But in production, manual setup means undocumented, unreproducible, and fragile infrastructure.

png18: 7 Terraform files — each one replaces manual gcloud commands

FileWhat It CreatesReplaces
main.tfProvider config + API enablementgcloud services enable ...
cloud_run.tfCloud Run service (scale to zero)gcloud run deploy ...
storage.tf3 GCS buckets + Artifact Registrygcloud storage buckets create ...
iam.tfService account + 3 permissionsgcloud iam service-accounts ...
monitoring.tfAlert policies + email channelManual console setup
variables.tfProject ID, region, emailHardcoded values
outputs.tfURLs, bucket names after applyCopy-paste from console

One command: terraform apply — creates all 13 resources. One command: terraform destroy — removes everything. Cost drops to ₹0 instantly. Infrastructure becomes version-controlled, auditable, and reproducible.
